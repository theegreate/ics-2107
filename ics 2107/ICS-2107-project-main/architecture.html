Web Architecture

Chapter 1: What it means

Web architecture is the conceptual structure of the World Wide Web. The WWW or internet is a constantly changing medium that enables communication between different users and the technical interaction (interoperability) between different systems and subsystems. The basis for this is different components and data formats, which are usually arranged in tiers and build on each other. Overall, they form the infrastructure of the internet, which is made possible by the three core components of data transmission protocols (TCP/IP, HTTP, HTTPS), representation formats (HTML, CSS, XML), and addressing standards (URI, URL). The term web architecture should be distinguished from the terms website architecture and information architecture.


Types of Web Architecture

The internet is a medium that is constantly changing and expanded by numerous developers, programmers and various consortia such as the W3C. However, the architectures used can be schematically distinguished.

Client-server Architecture

Initially, the web consisted of a two-tiered architecture: clients and servers. Clients and servers shared the tasks and services that the system was supposed to perform. For example, the client may request a service from the server; the server answers the request by providing the service. Three prerequisites must exist in the web architecture for the distributed application systems to communicate with one another:

  • Representation formats with a fixed standard: The most frequently used formats are HTML and CSS; or XML when machines communicate with one another.
  • Protocols for data transfer: HTTP (Hypertext Transfer Protocol) or HTTPS (Hypertext Transfer Protocol Secure) is used in the web. Other applications, such as mail servers, use SMTP (Simple Mail Transfer Protocol) or POP (Post Office Protocol). Determining the protocols used depends on the application.

  • The standard for addressing: This refers to the URL (Uniform Resource Locator) which is an instance of the more general concept of URI.

    Chapter 2: Centralised & Distributed Processing

    Centralised and distributed processing are two paradigms that define how a system handles computations and resources. Both play crucial roles in shaping the functionality and scalability of web systems.

    Centralised Processing

    In a centralised model, a single central unit, such as a server or mainframe, handles all computational tasks. This model simplifies management but comes with significant limitations.

    Advantages

    - Easier to manage and maintain as all operations occur in one location.
    - Stronger data consistency since there's only one source of truth.

    Disadvantages

    - Limited scalability due to reliance on a single processing unit.
    - Vulnerability to single points of failure.
    - Performance bottlenecks under heavy loads.

    Distributed Processing

    Distributed processing divides computational tasks across multiple systems, often connected through a network. This approach underpins modern web systems due to its scalability and resilience.

    Advantages

    - Supports larger user bases by scaling horizontally.
    - Ensures higher availability through redundancy and fault tolerance.
    - Enables faster response times by distributing resources closer to users.

    Disadvantages

    - More complex to synchronize and maintain consistency.
    - Requires sophisticated tools to manage communication between systems.

    These paradigms set the stage for the client-server model, where distributed processing forms the backbone of system interactions.

    Chapter 3: The Client-Server Model

    The client-server model is a distributed system design where clients (end-user devices) communicate with servers (backend systems) to access data and services. It is the core framework upon which web applications operate.

    Components

    Client:

    A device or application that initiates requests, such as a browser or a mobile app.

    Server:

    A system that processes client requests, retrieves the necessary data, and sends it back to the client.

    Workflow

    - A client submits a request (e.g., accessing a webpage).
    - The server processes the request, possibly interacting with databases or other services.
    - The server sends a response (e.g., an HTML page or JSON data).

    Advantages

      Decoupled Design

      The server sends a response (e.g., an HTML page or JSON data).

      Scalability

      Easily supports increased traffic by adding servers or balancing the load.

      Flexibility

      Supports a wide range of clients, from web browsers to IoT devices.

    By leveraging distributed processing, the client-server model can be optimized further through techniques like replication and specialisation, ensuring better performance and reliability.

    Chapter 4: Replication & Specialisation

    Replication and specialisation enhance the client-server model by addressing scalability, performance, and fault tolerance. These strategies are critical for modern web systems handling massive traffic and complex workloads.

    Replication

    Replication involves creating multiple copies of data or services to ensure availability and distribute workload.

    Data Replication

      - Stores identical datasets across multiple servers to improve read performance and reliability.
      - Enables quicker data access for geographically distributed users.

    Service Replication

      - Duplicates servers or applications to handle more requests and reduce latency.
      - Enhances fault tolerance by redirecting requests to operational replicas during failures.

    Specialisation

    Specialisation assigns specific tasks to dedicated components or systems for optimal performance.

    Dedicated Servers

      - Allocates individual servers for specific functions, like database management or caching.
      - Reduces resource contention and improves efficiency.

    Microservices

      - Breaks applications into smaller, independent services, each focusing on a specific business function.
      - Simplifies maintenance and scaling of individual components.

    Replication and specialisation lay the foundation for tiered web architectures, which structure web systems into layers for improved organization and performance.

    Chapter 5: Tiered Web Architectures

    Tiered architectures divide a web system into distinct layers or tiers, each responsible for specific functionalities. This modular design enhances maintainability, scalability, and fault isolation.

    Common Tiers

      Presentation Tier:

        - The user interface layer interacting directly with end-users.
        - Handles input collection and data display.

      Application Tier:

        - The business logic layer where computations and decision-making occur.
        - Processes client requests and manages interactions with the data tier.

      Data Tier:

        - The storage layer responsible for managing data retrieval, storage, and security.

    Benefits

      Independent Scaling:

      Each tier can be scaled based on specific demands, optimizing resource usage.

      Fault Tolerance

      Issues in one tier are isolated, preventing cascading failures.

      Flexibility

      Allows independent updates or replacements for tiers without disrupting the entire system.

    Tiered architectures are especially crucial in enabling advanced platforms like portal services, which aggregate various functionalities for seamless user experiences.

    Chapter 6: Portal Services

    Portal services are platforms that integrate multiple services and content into a unified interface, providing users with centralized access to diverse functionalities. They are a prime example of tiered architectures in action.

    Key Features

      Aggregation

      - Combines diverse services like emails, calendars, and news feeds into one interface.
      - Enhances productivity by reducing the need for multiple applications.

      Personalisation

      - Customizes content and services based on user roles, preferences, or behavior.
      - Increases user satisfaction through tailored experiences.

      Integration

      - Connects with external systems using APIs and protocols like OAuth for seamless data sharing.
      - Ensures interoperability across different platforms.

    Examples

      Enterprise Portals:

      Intranets like Microsoft SharePoint facilitate employee collaboration.

      Consumer Portals:

      Public-facing platforms like Google Workspace offer integrated tools for everyday tasks.

    Portal services showcase the culmination of replication, specialisation, and tiered architectures, delivering scalable, efficient, and user-centric platforms.
